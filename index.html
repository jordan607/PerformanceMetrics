01
First 5–10 minutes
Requirements Clarification
Never start drawing boxes until you've asked these. An interviewer wants to see that you scope the problem before solving it. Requirements define everything downstream.
Functional requirements — what the system must do
Core use case
What is the primary action users will perform?
Identifies the single most critical path. Design this first, then extend.
User types
Who are the actors? End users, admins, third-party services?
Different actors have different auth, access, and latency needs. Drives API surface.
Read vs write
Is the system read-heavy, write-heavy, or balanced?
Determines caching strategy, DB choice, replica configuration, and cost model.
Real-time
Does any feature require real-time or near-real-time updates?
Real-time = WebSockets / SSE / polling, not REST. Changes infra significantly.
Offline / async
Can any operations be asynchronous or eventually consistent?
Async paths unlock event queues, better throughput, and decoupled services.
Search
Is full-text search, geo-search, or faceted filtering required?
Signals need for Elasticsearch / PostGIS / Typesense — not just a DB index.
Media / files
Are images, videos, or binary files part of the system?
Triggers blob storage (S3), CDN, pre-signed URLs, and compression pipelines.
Notifications
Does the system need push, email, or SMS notifications?
Adds notification service, delivery guarantees, and unsubscribe/compliance concerns.
Payments
Are financial transactions in scope? Refunds, splits, subscriptions?
Triggers PCI DSS scope, idempotency, Saga pattern, and gateway integration.
Geographic scope
Single region, multi-region, or global deployment?
Multi-region changes latency targets, data residency, failover strategy, and cost 10×.
Non-functional requirements — constraints and quality attributes
Availability SLA
What is the required uptime? 99.9%, 99.99%, 99.999%?
Each "nine" costs exponentially more. 99.99% = 52 min downtime/year vs 99.999% = 5 min.
Latency target
What is the acceptable p99 response time for critical paths?
Drives caching, DB indexing, CDN placement, and what you can afford async.
Consistency
Strong or eventual consistency? What can tolerate stale reads?
Strong consistency is expensive at scale. Inventory / payments need it; feeds don't.
Durability
Can any data be lost? What is the acceptable RPO?
RPO = Recovery Point Objective. Zero data loss requires synchronous replication.
Security
What compliance standards apply? GDPR, PCI DSS, HIPAA, SOC 2?
Each standard mandates specific encryption, audit logging, and data handling controls.
Cost constraints
Is there a budget ceiling? On-prem vs cloud preference?
Cost changes every architecture decision — managed services vs self-hosted, storage tiers.
💡 Interview tip: Spend exactly 5 minutes here. Write your assumptions on the whiteboard. Say out loud: "I'll assume X, Y, Z — let me know if those are wrong." This shows senior-level thinking.

03
Define the contract
API Design
Define the interface before the internals. A well-designed API surface reveals the system's true scope and prevents scope creep mid-design.
Questions to ask about the API
Protocol
REST, GraphQL, gRPC, or WebSocket? Or a mix?
REST = simple CRUD. GraphQL = flexible queries. gRPC = high-perf internal. WS = real-time push.
Versioning
How will breaking changes be managed? /v1/, header versioning?
Without versioning, any change breaks clients. Define strategy early.
Idempotency
Are write endpoints idempotent? How are duplicate requests handled?
Critical for payments and bookings — retries must not double-charge or double-book.
Pagination
Offset-based, cursor-based, or keyset pagination?
Offset breaks on concurrent inserts. Cursor pagination is consistent for real-time feeds.
Rate limiting
Per user, per IP, per API key? What are the limits?
Prevents abuse and protects downstream services. Token bucket or sliding window.
Auth model
JWT, OAuth 2.0, API keys, mTLS for internal services?
External APIs use JWT/OAuth. Internal service-to-service uses mTLS or service mesh.
API Gateway responsibilities — always cover these
  • Authentication & JWT validation
  • Rate limiting & throttling
  • Request routing to services
  • SSL/TLS termination
  • Request / response transformation
  • Load balancing across instances
  • Canary & A/B routing
  • API versioning enforcement
  • DDoS protection & WAF
  • Observability (access logs, metrics)
  • Circuit breaker / timeout enforcement
  • Caching at edge
04
Choose the right store
Data Model & Storage
The database choice is often the most consequential architecture decision. Ask these questions before picking SQL vs NoSQL vs graph vs time-series.
Data model questions
Data structure
Is the data relational, hierarchical, graph-like, or schema-less?
Relational → RDBMS. Hierarchical docs → MongoDB. Graph → Neo4j. Schema-less → DynamoDB.
Query patterns
What are the top 3 most frequent query patterns?
Design schema around queries, not entities. Denormalize if necessary for read performance.
Joins
Are there frequent multi-table joins? Across large datasets?
Heavy joins at scale → consider denormalization, CQRS, or pre-computed views.
ACID requirements
Which operations need atomic, transactional guarantees?
Payments and inventory need ACID. User feeds and analytics can tolerate eventual consistency.
Time-series
Is any data append-only or time-indexed (metrics, events, logs)?
Time-series DBs (InfluxDB, TimescaleDB) are 10× more efficient for this pattern.
Hot vs cold data
What percentage of data is accessed frequently vs archived?
Tiered storage: hot → SSD, warm → HDD, cold → S3 Glacier. Cuts costs drastically.

Caching strategy questions
Cache-aside vs write-through
Should the app populate the cache, or the DB write through?
Cache-aside = lazy load on miss. Write-through = always consistent but higher write latency.
TTL strategy
What is the acceptable staleness? How long should each key live?
Short TTL = consistent, more DB hits. Long TTL = stale risk but better offload.
Cache eviction
LRU, LFU, or random eviction when cache is full?
LRU evicts least-recently-used. LFU evicts least-frequently-used. Depends on access pattern.
Cache invalidation
How is the cache cleared when source data changes?
"There are only two hard problems in CS: cache invalidation and naming things."
05
Draw the boxes
High Level Architecture
Now draw the system. Start with the happy path data flow, then add failure modes, scaling, and cross-cutting concerns.
Architecture pattern questions
Monolith vs microservices
What is the team size and deployment frequency?
Microservices pay off at scale (>50 engineers). For small teams, a modular monolith is faster.
Sync vs async
Which service calls must be synchronous? Which can be async?
Async (Kafka/SQS) decouples services, improves resilience, increases throughput.
CQRS
Should reads and writes have separate models / services?
CQRS separates read-optimized views from write-optimized models. Great for event sourcing.
Event sourcing
Should state be derived from an immutable event log?
Useful for audit trails, replaying state, and temporal queries. Higher complexity cost.
Service mesh
How will service-to-service communication be managed at scale?
Istio / Linkerd handles mTLS, retries, circuit breaking, observability between services.
Multi-tenancy
Are multiple customers isolated in the same infrastructure?
Drives schema design (shared vs isolated DB per tenant) and access control model.
Core architecture components — cover all in every design
  • Client (mobile / web / IoT)
  • CDN (static assets, edge caching)
  • DNS + load balancer
  • API Gateway (auth, routing, rate-limit)
  • Core microservices / application tier
  • Message queue / event bus (Kafka, SQS)
  • Cache layer (Redis, Memcached)
  • Primary database (OLTP)
  • Read replicas / secondary DBs
  • Object / blob storage (S3)
  • Search service (Elasticsearch)
  • Notification service (push, email)
  • Monitoring & alerting (Prometheus)
  • Logging (ELK / CloudWatch)
  • Distributed tracing (Jaeger / Zipkin)
  • CI/CD pipeline
06
Go deep
Component Deep Dives
Pick the 2–3 most interesting or risky components and go deep. This is where senior candidates differentiate themselves.
Scalability deep dive questions
Horizontal scaling
How does each service scale out? Are there stateful components?
Stateless services scale horizontally. Stateful (sessions, locks) need sticky routing or distributed state.
Database sharding
What is the sharding key? How are hotspots avoided?
Bad shard key = one shard gets all traffic. Good keys distribute writes evenly (hashed user_id).
Hotspot mitigation
What happens when a celebrity / viral entity gets 1M requests?
Famous users need special caching, read fanout strategies, or dedicated resources.
Auto-scaling
What metric triggers scale-out? CPU, RPS, queue depth?
CPU is a lagging indicator. RPS or queue depth are leading indicators — react faster.
Connection pooling
How are DB connections managed at high concurrency?
Without pooling, 1000 app threads = 1000 DB connections. PgBouncer / HikariCP solve this.
Backpressure
What happens when downstream services are overloaded?
Circuit breaker + backpressure prevents cascade failures. Fail fast > timeout storms.
Concurrency & consistency deep dive
Race conditions
What happens if two users try to book / buy the same resource simultaneously?
Distributed lock (Redis SETNX) or optimistic locking (DB version column) required.
Distributed transactions
How are multi-service operations made atomic?
Saga pattern (compensating transactions) preferred over 2PC which blocks under failure.
Idempotency keys
How are duplicate API calls detected and de-duped?
Client sends idempotency key → server stores result → replay returns same result, no double action.
Leader election
If a component needs a single active instance, how is the leader chosen?
ZooKeeper, etcd, or Redis-based leader election. Needed for schedulers, cron jobs, coordinators.
07
Cross-cutting concerns
Non-Functional Requirements
These are the "-ilities" — reliability, security, maintainability. They don't appear on the happy path but determine whether the system survives production.
Security questions — ask every time
Authentication
How are users authenticated? Token lifetime? MFA?
Short-lived JWTs (15 min) + refresh token rotation. MFA for high-value actions.
Authorization
RBAC, ABAC, or ReBAC? How are permissions enforced?
RBAC = roles. ABAC = attribute-based. ReBAC = relationship-based (Google Zanzibar).
Encryption
TLS in transit, AES-256 at rest? Field-level encryption for PII?
TLS 1.3 for transport. AES-256 for storage. Field-level for SSN, card numbers, health data.
Secrets management
How are DB credentials, API keys, and certs managed?
Vault / AWS Secrets Manager. Never in code. Rotate automatically. Audit on access.
Input validation
How is SQL injection, XSS, and SSRF prevented?
Parameterized queries, CSP headers, allowlisting outbound requests. WAF at edge.
Audit logging
What actions must be logged for compliance? Who can read the logs?
Immutable append-only audit trail. CloudTrail + tamper-evident storage for compliance.

08
Keep it running
Ops, Reliability & Observability
A system that can't be observed can't be operated. Design for failure from the start — not as an afterthought.
Reliability questions
Fault tolerance
What happens when any single component fails?
No single point of failure. Every service should have ≥2 replicas across ≥2 AZs.
Circuit breaker
How does the system stop cascading failures?
Circuit breaker opens on repeated failures, fast-fails callers, gives downstream time to recover.
Retry strategy
Exponential backoff with jitter for all retries?
Synchronized retries amplify load. Jitter spreads retries, reducing thundering herd.
Graceful degradation
Which features can be disabled to protect core functionality?
Feature flags let you shed non-critical load under stress. Core path must always work.
Disaster recovery
What is the RTO and RPO? Active-active or active-passive?
RTO = time to recover. RPO = data loss tolerance. Active-active costs 2× but near-zero RTO.
Chaos engineering
How are failure scenarios validated in production?
Chaos Monkey, fault injection. Discover failures in drills, not during incidents.
Observability pillars — the three must-haves
📊 Metrics
Prometheus / Grafana / CloudWatch. Track RPS, p99 latency, error rate, CPU, memory. Set alerts on SLO burn rate. Use RED method (Rate, Errors, Duration) per service.
📋 Logs
Structured JSON logs. ELK stack or Datadog. Correlation IDs across services. Log levels: ERROR (page on-call), WARN (alert), INFO (operational), DEBUG (local only).
🔍 Traces
Jaeger / Zipkin / AWS X-Ray. Trace ID propagated across all service hops. Shows exactly which service added latency to a slow request. Critical for debugging microservices.
Deployment strategy questions
Deployment pattern
Blue-green, canary, rolling, or feature flags?
Canary: 1% → 10% → 100% with automatic rollback on error rate spike. Safest for production.
Schema migrations
How are DB schema changes deployed without downtime?
Expand-contract pattern: add new column (expand), migrate data, drop old column (contract). Never both in one deploy.
09
Show wisdom
Trade-offs & Alternatives
The final step separates senior candidates from mid-level ones. Acknowledge that every decision has a cost. Show that you made deliberate choices, not defaults.
Core system design trade-offs
Consistency vs Availability
CAP theorem: under network partition, choose one. AP = always available, maybe stale (DNS, shopping carts). CP = consistent, may reject requests (banking, inventory).
Latency vs Throughput
Batching increases throughput but adds latency. Real-time responses have lower throughput. Choose based on user experience requirement — interactive vs background processing.
Normalisation vs Denormalisation
Normalized = less storage, complex joins, slower reads. Denormalized = fast reads, storage overhead, complex writes. Denormalize for read-heavy hot paths.
Push vs Pull
Push (fan-out on write) = fast reads, expensive writes for high-follower users. Pull (fan-out on read) = cheap writes, expensive reads. Hybrid: push for regular users, pull for celebrities.
SQL vs NoSQL
SQL = ACID, complex queries, vertical scale, relational integrity. NoSQL = horizontal scale, flexible schema, eventual consistency. Most real systems use both.
Monolith vs Microservices
Monolith = simpler ops, single deploy, shared memory. Microservices = independent scale/deploy, polyglot, distributed tracing overhead. Don't split until you feel the monolith's pain.
Final checklist — before you close the design
  • Single points of failure eliminated
  • Scale path defined (3× and 10× growth)
  • Data consistency model stated
  • Authentication & authorization covered
  • Encryption at rest and in transit
  • API rate limiting in place
  • Circuit breakers and retries defined
  • Monitoring / alerting / tracing mentioned
  • DB backup and recovery plan
  • Deployment strategy (zero downtime)
  • Trade-offs explicitly acknowledged
  • Bottleneck identified and addressed
⚠️ The one thing interviewers test for: Do you know why you made each choice? "I used Kafka because we need async processing and at-least-once delivery with replay capability" beats "I used Kafka because it's popular." Every choice should have a specific justification tied to your requirements.

03 · formulas

The core formulas — all of them

Every back-of-envelope estimation in a system design interview reduces to one or more of these formulas. Learn them as patterns, not as algebra.

⚡
Read RPS
Daily active users × reads per user per day, divided by seconds in a day.
RPS = DAU × reads/day ÷ 86,400
✍️
Write TPS
Same formula as reads, but for write events (posts, purchases, messages).
TPS = DAU × writes/day ÷ 86,400
📦
Daily new data
Writes per day multiplied by bytes per object. This is your daily storage growth.
Daily = TPS × 86,400 × bytes/object
🗄️
Total storage (N years)
Daily growth times days times replication factor. Always include replication — disk cost is 3× your data.
Total = Daily × 365 × years × repl_factor
📡
Bandwidth (outbound)
Reads per second times bytes per response. Convert to Mbps by multiplying by 8 and dividing by 10⁶.
Mbps = RPS × response_bytes × 8 ÷ 10⁶
🔥
Peak traffic
Always provision for 2–5× the average. Never size for average — you'll collapse under any spike.
Peak_RPS = Avg_RPS × peak_multiplier
🧠
Cache size (80/20 rule)
20% of your content gets 80% of traffic. Size your cache to hold that 20% hot data.
Cache = Total_objects × 0.20 × bytes/object
🖥️
Server count
Peak RPS divided by what a single server can handle. Add ~30% headroom so you never run at 100%.
Servers = Peak_RPS ÷ RPS_per_server × 1.3
🔌
Concurrent connections (Little's Law)
Connections = Arrival rate × average time a request is "in flight". Latency kills concurrency.
Conc = RPS × avg_response_time_seconds
📊
QPS to DB (read)
Reads per second × (1 − cache hit rate). This is the DB load after the cache absorbs its share.
DB_QPS = RPS × (1 - cache_hit_ratio)
🌐
CDN origin load
With a CDN, the origin only serves cache misses. A 95% hit rate means the origin sees 5% of raw traffic.
Origin_RPS = RPS × (1 - cdn_hit_rate)
🔐
Fan-out writes (social)
For social feeds: 1 post by a user with N followers = N writes to timelines. Viral posts cause write explosions.
Timeline_writes = Posts/s × avg_followers
04 · process

Step-by-step method — in order, every time

The estimation phase of a system design interview has a repeatable structure. Follow it in the same order every time so it becomes automatic. The sequence matters — each step's output feeds the next.

01
State your assumptions out loud, write them on the board
Before touching a number, declare: "I'll assume 100 million DAU. I'll assume a 9:1 read-write ratio. I'll assume average post size is 500 bytes. Tell me if any of these are off." This shows senior-level thinking — you know numbers come from assumptions, not from the sky. The interviewer may correct you; that's fine, that's the point.
Assumptions: DAU = 100M Read:Write = 10:1 Avg post = 500 bytes (text) + 2 MB (if image, separate) Retention = 5 years Replication = 3x (standard)
02
Calculate traffic — reads, writes, and peak
Convert DAU × actions/day into requests per second. Do reads, then writes separately. Then apply your peak multiplier — design for peak, not average. Round aggressively — "roughly 23K, let's call it 25K."
Read RPS = 100M × 20 reads/day ÷ 86,400 = 23,148 ≈ 25K RPS Write TPS = 100M × 2 writes/day ÷ 86,400 = 2,315 ≈ 2.5K TPS Peak reads (3×) = 75K RPS Peak writes (3×) = 7.5K TPS
03
Calculate storage — daily growth, then 5-year total
Daily new data = write TPS × seconds/day × bytes/record. Then project for your retention period. Always multiply by replication factor at the end — the real disk cost is 3× your logical data size.
Daily new data = 2,500 writes/s × 86,400 s × 500 B = 108 GB/day 5-year logical = 108 GB × 365 × 5 = 197 TB With 3× repl. = 197 TB × 3 ≈ 600 TB total disk Note: image uploads handled separately → S3, not DB
04
Calculate bandwidth — outbound, then inbound
Outbound bandwidth = peak read RPS × average response size × 8 (to bits). Inbound = write TPS × average request size × 8. Compare these to your chosen infrastructure's network limits.
Outbound = 75,000 RPS × 5,000 bytes × 8 ÷ 10⁶ = 3,000 Mbps = 3 Gbps Inbound = 7,500 TPS × 500 bytes × 8 ÷ 10⁶ = 30 Mbps → Outbound is the bottleneck — need multi-Gbps egress, CDN is essential
05
Calculate memory / cache size
Apply the 80/20 rule: 20% of objects serve 80% of reads. Size your Redis cluster to hold that 20% hot data. Also account for session data, distributed locks, and any other in-memory state.
Total objects = 2,500 writes/s × 86,400 × 365 × 5 = 394 billion records Hot 20% = 79 billion records × 500 B = 39 TB (too large for RAM) → Strategy: cache recent 7-day data (hot window) 7-day objects = 2,500 × 86,400 × 7 = 1.5 billion × 500 B = 756 GB Redis cluster ≈ 1 TB RAM (with headroom)
06
Calculate server count and DB shard count
Divide peak RPS by a single server's capacity (typically 1K–10K RPS depending on the operation). Add 30% headroom. For the DB, if writes exceed ~5K TPS for a single primary, you need sharding — divide total TPS by what a single shard can handle.
App servers = 75,000 peak RPS ÷ 5,000 RPS/server × 1.3 = 20 servers (min) DB primaries: 7,500 write TPS ÷ 3,000 TPS/shard = 3 shards min DB replicas : 3 shards × 2 replicas each = 6 read replicas Total DB nodes: 9
07
Derive your architecture decisions from the numbers
This is the step most candidates skip. The numbers aren't the end — they tell you which components you need. High read RPS → caching layer is mandatory. High write TPS → event-driven async. Large data → sharding. High bandwidth → CDN. Let the estimates justify your design choices out loud.
→ 25K read RPS: Redis cache mandatory (DB alone can't handle it) → 2.5K write TPS: within single PostgreSQL primary — no sharding yet → 3 Gbps outbound: CloudFront CDN mandatory, images to S3 → 600 TB total: object storage (S3) + RDBMS for hot data + archive cold → 5-year: implement data tiering (hot → SSD, cold → S3 Glacier)
05 · interview technique

How to run this in a live interview

The content of your estimation matters less than whether you demonstrate a methodical, collaborative process. Here's the exact sequence interviewers at top companies expect to see.

min 0–2
Ask for the scale, don't assume it
"Before I estimate, can I ask a few clarifying questions? How many users are we designing for — are we thinking startup scale, or something like Twitter at 300 million DAU?" The interviewer will often give you a number or say "you tell me" — both are fine. If they say "you decide," pick a round number and say it out loud.
min 2–5
Write your assumptions first, before any math
Write the raw inputs on the board: DAU, read:write ratio, object size, retention. Say each one as you write it. This gives the interviewer a chance to correct you before you build a skyscraper on a wrong foundation, and shows you understand that estimates start with assumptions.
min 5–10
Work through traffic → storage → bandwidth in that order
Don't jump around. The order matters: traffic tells you your compute needs, storage tells you your database needs, bandwidth tells you your network needs. Talk as you calculate — "2 billion reads divided by 86,400 seconds, that's roughly 23K reads per second, let's round to 25K." The interviewer is watching your reasoning, not grading your arithmetic.
min 10–12
Derive conclusions — don't just dump numbers
"So at 25K read RPS, a single database can't handle this — we'll need a caching layer. At 600 TB over 5 years, we'll need object storage, not just a relational DB. At 3 Gbps outbound, we need a CDN." This is what separates senior engineers: you use the numbers to justify the architecture, not just to show you can do arithmetic.
min 12+
Keep the estimates visible — refer back to them
Don't erase your estimates when you start drawing the architecture. When you place the cache, say "because of our 25K RPS requirement." When you add sharding, say "because at 2.5K write TPS we're approaching the limit of a single primary." The numbers become the design's justification throughout the interview.
The single most common mistake: candidates estimate and then ignore the numbers when drawing the architecture. The estimates exist to drive decisions — reference them every time you add a box. "I'm adding Redis here because our 25K read RPS would saturate a database primary" is a senior answer. Just drawing a Redis box is a junior answer.
06 · what to avoid

Common mistakes — bad vs good

These are the actual patterns that cause otherwise solid candidates to lose seniority signals during estimation.

❌ Bad
"Let's say 1 million users." (no justification, no derivation)
✓ Good
"I'll assume 10M DAU — that's in the range of a mid-scale product. If this is meant to be Twitter-scale I'll adjust."
❌ Bad
"We'll need about 1 TB of storage." (no calculation shown)
✓ Good
"10M users × 10 writes/day = 100M writes/day. × 500B = 50GB/day. × 365 × 5 = ~90TB. × 3 replication = ~270TB total."
❌ Bad
Spending 20 minutes on estimation, producing a spreadsheet of exact numbers
✓ Good
10 minutes max. Round to nearest power of 10. "~25K, let's call it 30K" is correct interview math.
❌ Bad
Erasing the estimates and drawing the architecture from memory without connecting the two
✓ Good
Keep estimates visible. Reference them every time you add a component: "Cache because of 25K RPS."
❌ Bad
Forgetting to multiply by replication factor for storage — getting the number 3× too small
✓ Good
Always: logical_data × replication_factor (3). Real disk cost is always 3× what you store logically.
❌ Bad
Sizing for average traffic, not peak — your system collapses on the first big event
✓ Good
Always apply a 2–5× peak multiplier after calculating average. Provision for peak. Autoscale down to average.
❌ Bad
Not accounting for the 80/20 rule on cache — sizing cache for all data instead of just hot data
✓ Good
"20% of objects serve 80% of reads — I'll size the cache to hold that 20% hot slice, not everything."
❌ Bad
Forgetting media (images, video) — under-estimating storage by 100× for image-heavy products
✓ Good
"Text: 500B/post. But if users can upload photos (~2MB avg), that's 4,000× more storage — handle separately with S3."
07 · worked examples

Twitter / X — full estimation walkthrough

Read this as if you were speaking it out loud in an interview. Every line is a thought spoken, not just a calculation.

// system: Twitter / X — read-heavy social media with fan-out writes target: ~10 min
STEP 1 — ASSUMPTIONS
DAU=300Mstated in problem
Tweets per user per day≈0.1most users read, few post
Timeline reads per user per day≈50scrolls through feed
Avg tweet size (text)=280 bytes ≈ 300 Bchar limit
Avg followers per user=200geometric mean, not arithmetic
Retention=5 yearsassumption
STEP 2 — TRAFFIC
Total tweets/day= 300M × 0.1= 30M tweets/day
Write TPS (tweets)= 30M ÷ 86,400≈ 350 TPSvery low!
Timeline reads/day= 300M × 50= 15B reads/day
Read RPS= 15B ÷ 86,400≈ 175K RPSvery high!
Read:Write ratio= 175K ÷ 350≈ 500:1extreme read skew → cache everything
Fan-out writes/s= 350 TPS × 200 followers= 70,000 fan-out writes/scelebrity posts explode this
Peak (3×)525K read RPS, 210K fan-out TPS
STEP 3 — STORAGE
Daily tweet data= 30M × 300 B= 9 GB/day text
5-year text data= 9 GB × 365 × 5≈ 16 TBtiny — tweets are small
With 3× replication= 16 TB × 3≈ 50 TB
Media (assume 10% tweets have image, 2MB)= 3M images/day × 2MB= 6 TB/day ← dominant cost
5yr media total= 6 TB × 365 × 5≈ 10 PB → S3, not DB
STEP 4 — BANDWIDTH
Read bandwidth= 525K RPS × 5KB/response × 8= 21 Gbps outboundCDN mandatory
Write bandwidth= 350 TPS × 300 B × 8≈ 1 Mbps inbound
STEP 5 — CONCLUSIONS → ARCHITECTURE DECISIONS
175K read RPS→Redis timeline cache — DB alone can't serve this
500:1 read:write→Push fan-out (pre-write timelines) for most users
70K fan-out/s→Kafka for async fan-out — synchronous would time out
Celebrity posts→Pull model for users with >1M followers (hybrid approach)
10 PB media→S3 + CloudFront CDN, not database storage
07b · worked example

YouTube — video-heavy read-dominated system

// system: YouTube — bandwidth and storage dominated by video target: ~10 min
ASSUMPTIONS
DAU=2 billion (MAU) → ~500M DAU
Video watched per user per day≈30 min = 5 videos avg
Videos uploaded per day≈500K (0.1% of users upload)
Avg video size (raw)=1 GB (10 min at 4K before encoding)
Encoded sizes=4 resolutions × 200 MB avg = 800 MB per video
TRAFFIC
Video views/day= 500M × 5= 2.5 billion views/day
Streaming RPS= 2.5B ÷ 86,400≈ 29K concurrent streams
Upload TPS= 500K ÷ 86,400≈ 6 uploads/slow count, huge size
STORAGE & BANDWIDTH
New storage/day= 500K × 800 MB= 400 TB/daymassive!
5-year storage= 400 TB × 365 × 5≈ 730 PB ≈ 1 exabyte (with repl)
Streaming bandwidth= 29K streams × 5 Mbps avg= 145 Gbps outbound
CONCLUSIONS
400 TB/day upload→Distributed object storage (GCS/S3) with async encoding pipeline
145 Gbps streaming→Global CDN with edge caching is non-negotiable
1 exabyte→Tiered storage: hot (SSD), warm (HDD), cold (tape/Glacier)
Video encoding→Separate encoding farm (GPU workers) — not same service as serving
07c · worked example

Uber — real-time geolocation and matching

// system: Uber — location updates dominate all other traffic target: ~10 min
ASSUMPTIONS
Active drivers (at any moment)≈1 million globally at peak
Location update frequency=every 4 seconds (GPS ping)
Active riders requesting≈5M users browsing for a ride at peak
Avg trips/driver/day=15 trips
Location update payload=100 bytes (lat, lng, driver_id, ts)
TRAFFIC — LOCATION IS THE BOTTLENECK
Location writes/s= 1M drivers ÷ 4s= 250,000 writes/sthis is huge
Rider nearby-driver queries= 5M × 1 query/10s= 500K geo queries/s
Trip creation TPS= 1M drivers × 15 trips ÷ 86,400≈ 174 TPStiny vs location
STORAGE
Location log/day= 250K/s × 86,400 × 100B= 2.16 TB/daybut only hot window matters
Hot state (drivers now)= 1M drivers × 100B= 100 MBtiny! fits in RAM easily
Trip history (5yr)= 174 TPS × 86,400 × 365 × 5 × 500B≈ 12 TB
CONCLUSIONS
250K location writes/s→Cassandra or Redis Geo — PostgreSQL at 5K TPS can't handle this
100MB hot state→All active driver positions fit in a single Redis instance
500K geo queries/s→Geohash-partitioned in-memory lookup, not DB query
Websockets→Persistent connections for driver apps (not polling) — 1M open sockets
07d · worked example

WhatsApp — high write TPS, delivery guarantees

// system: WhatsApp — 100B messages/day, delivery critical target: ~10 min
ASSUMPTIONS
DAU=2 billion
Messages/user/day=50 (send + receive combined)
Avg message size=100 bytes text (media handled separately)
Group chat: avg group size=30 members, 20% of messages
TRAFFIC
Messages/day= 2B × 50= 100B messages/day
Message TPS (write)= 100B ÷ 86,400≈ 1.16M msg/smassive write volume
Group fan-out= 1.16M × 20% × 30 members= 6.96M deliveries/s
Peak (2×)2.3M msg/s write, 14M deliveries/s
STORAGE
Message storage/day= 100B × 100B= 10 TB/day
Note: WhatsApp stores on device→server only keeps for delivery, then deleteskey design choice
Undelivered message buffer= assume 7-day max hold= 70 TB buffer
CONCLUSIONS
1.16M msg/s→Kafka ingestion + Cassandra for message store (high write throughput)
Delete after delivery→Storage stays bounded despite huge volume
14M deliveries/s→Persistent WebSocket connections per device, connection server fleet
Delivery guarantee→At-least-once with dedup (idempotency keys), acked by device
Cross-example pattern: in every worked example, the surprising insight comes from the numbers, not from prior knowledge. Uber's insight is that 1M driver locations fit in 100MB of RAM — you can serve all geo queries from a single Redis node. Twitter's insight is the 500:1 read:write ratio demanding pre-computed timelines. WhatsApp's insight is that storing on-device instead of server-side collapses a 70TB storage problem. The math tells you the architecture.
